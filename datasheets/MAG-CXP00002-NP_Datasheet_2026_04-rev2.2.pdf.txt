The MAG -CXP00002 -NP is a camera interface ASIC that 
support s harsh environment video transmission  over a 
75-Ohm coax cable  for camera applications.  
The MAG -CXP00002 -NP is a radiation -hardened  device -
side CoaXPress  2.0 video transmission ASIC. The ASIC 
supports downlink video streaming at 1.25Gbps  (CXP -1 
mode)  and receives uplink control commands at 
20.83Mbps , all over a single 75 -Ohm cable. In addition, 
it also supports Power -over -Coax (PoC) and is 
compatible with COTS CXP frame grabbers. For CXP 
certified cables, the cable length can go above 100m in 
the mode of operation  implemented (CXP -1). 
The parallel input interface supports raw parallel pixel 
data from an image sensor with up to 12b pixel depth 
and is capable of interfacing different frame sizes and 
frame rates.  
MAG -CXP00002 -NP features a dedicated master SPI 
interface to  communicate with an image sensor  chip , as 
well as a secondary master SPI interface , allowing 
communication to other  SPI slave devices, such as chips 
from the Magics Motion Series . The serializer ASIC also 
features imager control pins such as a RESET  signal  and 
a TRIGGER  signal  and integrate s perfectly  with the 
MAG -IMG001X1 -NC image sensor of Magics. The chip 
set can form the heart of a camera system  and the 
combination is fully GenICam  compliant.  From a user 
point of view (e.g. CXP host) the chip set is seen as a 
single device , because of the integrated register map 
bootstrapping implemented in the serializer ASIC , 
which holds a copy of the register map of the imager.

A simplified pinout diagram of  MAG -CXP00002 -NP can be seen in  Figure 2 . The connection to the camera chip consists 
of the 12 pins that receive data from the imager  chip (VIDEO_DATA<11:0>) and the synchronization signals  (i.e. 
PIXEL_CLK_CAM, DATA_VALID_CAM, LINE_VALID_CAM  and FRAME_VALID_CAM). The serializer has a master SPI 
dedicated for the imager , in addition to a reset signal RESET_CAM and trigger signal TRIGGER_CAM to control  the imager 
chip. On the top row of the pins, there are 11 GPIOs pins that the user can  used  in different ways , such a s for a second 
master SPI interface . The connection to the external clock crystal or the clock reference and the coaxi al cable leading to 
the frame grabber are on the right side  of the chip  (XTALP/N, REF_CLK, TRXP/N). A 1.8V regulated power supply 
VDD1V8_CAM and a 1.2V reference voltage VREF_CAM can be used to provide power and a voltage reference to the 
imager , depending on the needs of the imager chip. Note that the VDD1V8_CAM pin cannot be used to drive the 1.8V 
domain of the MAG -IMG001X1 -NC imager, as it cannot supply enough current.  
The ASIC can be powered from a single 3.3V supply and power is given through multiple 3.3V pins. As there are several 
on-chip LDOs, external decoupling is required to compensate the loop for these LDOs. Therefore, there are multiple 
LDO pins on the chip as well.  
5. Block Diagram  
The block diagram in Figure 3  shows the main functional blocks of MAG -CXP00002 -NP. The VTOI, PTAT, VREF and IREF 
form the biasing block, providing all other blocks with reference voltages and currents. The LDO_CAM and CAM_REF 
provide a 1.8V power supply and a 1.2V reference voltage for the imager chip connected to MAG -CXP00002 -NP. The 
clock reference block, containing an oscillator and buffer for an external clock reference form together with the PLL the 
clock generation block, generating the required 125MHz and 1.25GHz on -chip clocks for the digital block and the 
serializer. The serializer, low-voltage differential signalling  (LVDS ) driver and the hybrid block form the front -end block  
which is  responsible for serializing the parallel data from the digital block and transmitting i t to a coaxial cable . In 
addition, it takes care of receiving  the slow uplink communication data sent by the frame grabber to the serializer  via 
the same  coax  cable. Finally, the digital block is responsible for deserializing the uplink communication, and the encoding 
of the parallel data from the imager to be sent to the serializer. The digital block also houses SPI modules for slave and 
master SPI interfaces , next to the GenICam  compliant register bank.

8. Detailed description  
In the next sections the operation of the device together with the different features is described in more detail.  
8.1 Power -on-sequence  
The chip can be powered from a single 3.3V supply. After pow ering the chip and after releasing the reset , a device 
discovery is  typically  initiated by the CXP h ost (i.e. frame grabber)  and consists of the following tasks:  
• Match Device transmitter and Host receiver bit rates to obtain register access.  
• Discover Devices and the connection topology.  
• Negotiate the CoaXPress  version to use  (CXP v2.0 is implemented, but it is compatible with lower versions) . 
• Detect the high speed up -connection (not supported in this version of the standard  – fixed at 20.83Mbps ). 
• Negotiate maximum packet sizes.  
• Set the connection speed after reading ConnectionConfigDefault  . 
Any link reset issued either in this device discovery sequence or by the user  (Address 0x4000)  triggers the bootstrap 
FSM module  and resets the connected imager for a selected delay.  The bootstrap FSM module is responsible for 
synchronizing the registers of MAG -CXP00002 -NP and the connected imager. From a user point of view, it seems that 
the chip set (imager and serializer) is a single device, sharing a single register map.  More information can be found in 
section 9.2 when used in combination with the Magics Full HD imager MAG -IMG001X1 -NC. 
 
8.2 Parallel v ideo  input  interface  
 
Figure 5: Video interface for MAG -CXP00002 -NP.

8.3 CXP interface  
 
Figure 7: CXP interface between MAG -CXP00002 -NP and a frame grabber . 
 
The CXP interface is depicted in Figure 7 . The device side, consisting of the imager  chip and the serializer MAG -
CXP00002 -NP is connected to the host side (frame grabber) via a 75 Ohm coaxial cable according to the CoaXPress  2.0 
standard.  As the LVDS driver of MAG -CXP00002 -NP has a differential output, only one output pin, TRXN is connected to 
the coaxial cable, while the other output pin is locally terminated on the PCB using a 75 Ohm resistance. The decoupling 
capacitors Cd are chosen to the values provided by the CoaXPress  2.0 standard documentation according (56nF in our 
case). If required, P oC can be applied on TP3 to supply  the device side with power. In this case, a P oC filter is also added 
on TP2, while the serializer chip should switch to the baseline wandering filtered uplink mode to correctly read the data 
from the host side.  Note that commercial frame grabbers always implement such PoC feature on the host side. However, 
the default PoC voltage defined by the standard is 24V. Note that Magics’ DC/DC converters (MAG -PSU00001 -NP) 
cannot handle such high voltage and are limited to  11V max at their input.  
8.4 Master SPI  interface  
Two serial Peripheral interface ( SPI) Masters  are provided where the first one is connected to the imager ’s slave 
interface for synchronizing MAG -CXP00002 -NP registers to the imager's registers and program its registers through 
uplink commands sent from the frame grabber. The  second SPI master is sharing the GPIO pins when enabled and is 
used to control an external ASIC having an SPI slave interface.  
The main features of the instantiated SPI master are:  
− The SPI mode is fixed to mode 0, other modes are not supported. This means:  
o The idle clock level is 0  
o Serial data is captured on the rising edge

8.4.1 Imager Master SPI  
This SPI master is dedicated to the imager connected to it. The control of the master is taken  care of  by the bootstrap 
FSM, which ensures that all configuration data ends up at the imager, and that all necessary information to configure 
itself, is read from the imager. The latter ensures that the serializer works together properly with the imager . More 
information can be found in section 9.2. 
For the  SPI master connected to the imager, the default clock frequency is 20.83Mhz. However, the frequency can be 
adapted through the register ‘VendorUniqueBootstrapFSM_spi_master_clock_divider ’. 
8.4.2 General purpose Master SPI  
As mentioned above, a second master SPI is implemented on the chip, and the IO pins  for the  hardware  interface are 
shared with the GPIO pins. The GPIO pins to be used for the general purpose master SPI are listed in section 6, i.e. pins 
58 to 61  (SPI_CSB0, SPI_CLK, SPI_MOSI, SPI_MISO) . The rest of the GPIO pins could also be used as SPI chip selects, 
which means that you can connect up t o 8 different slave SPI devices to the GPIO pins.  To enable the master SPI 
functionality on the GPIO pins, please set register bit ‘VendorUniqueSPIGPIO_spi_enable’. Note that also the clock 
frequency of the general purpose master SPI can be selected, but by the register ‘VendorUniqueSPIGPIO_divider’ . 
Default frequency is 62.5  MHz , because the ‘VendorUniqueSPIGPIO_d ivider’ value is set to a default 0 . 
The transaction size of the SPI command can be chosen with the register ‘VendorUniqueSPIGPIO_transaction_size’  and 
can be up to 64 bits.  The slave to be selected is to be  defined  in register ‘VendorUniqueSPIGPIO_ssel’ . The data to be 
sent to the slave needs to be written in ‘VendorUniqueSPIGPIO_data_mosi’  (note: 2 32 -bit registers – address 0xd440 
and 0xd44 8). The data that will be received from the SPI is to be found back in register  
‘VendorUniqueSPIGPIO_data_miso’  (note: 2 32 -bit registers – address 0xd 44c and 0xd450 ). If a SPI command needs to 
be initiated, the above settings need to be set before initiating the SPI access. The latter can be done by writing a 1 to 
the register ‘VendorUniqueSPIGPIO_trigger’ . This field is automatically reset to 0  afterwards . To monitor  if the 
transaction is still busy, the register ‘VendorUniqueSPIGPIO_busy’  could be read  and equals 1 if a transaction is busy.  
Note that this general purpose Master SPI is to be controlled through CXP up -link commands and as a result, the 
remaining bandwidth for these commands depend on the selected video mode and corresponding frame rate and data 
load . This means that the rate/speed of writing and reading SPI data can depend on the video mode selected. As a result, 
Magics advises to not use this SPI interface to be integrated in control loops which require accurate timing, such as 
motor control loops. For the latter, Magics advise s to use the BiSS communication chips from the Motion Series.  
8.5 GPIO s and programmability  
The GPIOs implementation is compliant with the GenICam /IIDC2 standard  (see Figure 8 ). The GPIO pins are also shared 
with an SPI master interface . The SPI master functionality is described in the previous section. This section covers the 
GPIO control. As mentioned, Figure 8  below gives an overview of the block diagram of the normal DigitalIOControl for 
each I/O line.  However, we do not implement all features as are depicted in the figure. Referring to Figure 8, we do not 
implement the TriggerSource  and TimerTriggerSource  blocks.  
By default,  all the GPIOs are set to input.  This could be changed  to an output  in register  ‘IOLineModeAll_Value ’. In 
addition, the ‘ IOLineSource_Value ‘ register needs to be set to 32 when a GPIO is set as an output. The polarity of the 
GPIO could be changed by setting the register ‘IOLineInverterAll_Value’ to 1. Note that each of these registers  are 11 bit 
wide to control each corresponding GPIO.   
To read the input from a GPIO, please refer to the register ‘IOLineStatusAll_Value’ . To drive the output of a GPIO pin, 
please refer to register ‘IOUserOutputValueAll_Value’.  Additionally, there is the functionality to connect internal source 
signals to an IO line . Please refer to registers ‘IOLineSelector_Value ’ and ‘IOLineSource_Value’  for this functionality.

8.7 GenICam  compliance  and bootstrap FSM  
The register map of MAG -CXP00002 implements the main bootstrap registers required from the CXP ( /GenICam ) 
standard  in addition to  all the other registers needed for analog configuration and trimming . The latter are not accessible 
by the customer and will be programmed by Magics by means of on -chip OTP eFuses.  Next, t he extra interfaces of the 
digital blocks  such as the GPIOs  are implemented according to the IIDC 2 standard wherefrom the entry point can be 
found at the address 0x1C. Section 11.1  gives an overview of all the registers used in the ASIC  which are accessible to 
the customer .  
The on -chip bootstrap FSM is responsible for the synchronization of the important settings between the serializer ASIC 
and any external imager that is compliant with  the GenICam  standard (e.g. MAG -IMG001X1 -NC). This guarantees an 
alignment between configuration settings such as image size, image format pixel format,  mirroring features, Bayer 
pattern configuration,  etc. The bootstrap FSM takes care of transferring the right configuration data to the externally 
connected imager, while also reading back the se ttings to be used in his own register. This is done by means of the 
dedicated SPI master. It also means that the user seems to be communicating to one device only and is blind for all the 
action happening in the bootstrap FSM.  Note that GenICam  compliance of the image sensor is mandatory to have the 
mechanism functioning correctly.  More information on how this copying mechanism works in application when an 
imager is connected, can be found in section 9.2. 
8.8 Debug SPI slave interface

It is a 96 -bits SPI transaction that is o perating in mode 0, which means: the serial data is captured on the rising edge and

9. Typical applications  
9.1 Application with a generic camera chip  and a frame grabber  
MAG -CXP00002 -NP is a radiation tolerant camera interface. The following design is an exemplary application of MAG -
CXP00002 -NP. 
 
Figure 9: Application scenario for the MAG -CXP00002 -NP. 
 
The MAG -CXP00002 -NP provides the connection between the frame grabber and the camera chip. All captured  image 
data is sent in a parallel bus to the serializer, along with the frame/line/data valid signals required to proper acquisition  
of individual pixel/line or entire frame data. Along with these signals, the pixel clock used in the camera chip is also se nt 
to the serializer to provide a means for data sampling. According to the GenICam  standard, both chips are treated as a 
single device from the point of view of the user and thus both can be programmed through a single uplink channel that 
shares the same coaxial cable as the downlink communication between the serializer and the frame gr abber. The 
properties of uplink , downlink , driver  and hybrid circuits  that handle them are set according to the CXP 2.0 standard. 
Serialized and encoded video data is sent via the coaxial cable to the frame grabber, while  any configuration of either 
the MAG -CXP00002 -NP serializer or the camera chip is sent by the frame grabber via the same coaxial cable to the 
serializer chip. When necessary, the device side can also be powered using the same coaxial cable as the one used for 
up- and downlink.

9.1.3 Connection to the host  
The connection between the MAG -CXP00002 -NP and the frame grabber is a coaxial cable with a characteristic resistance 
of 75 Ω according to the CoaXPress  2.0 standard. This cable is connected to pin TRXN, while TRXP is terminated locally 
on the PCB using a 75 Ω resistance . In case the P oC option is used , DC voltage sources and P oC filters can be connected 
at the host and the device side, respectively. More information on this can be found under section 8.3.  
9.1.4 Connection to the clock  
The clock reference circuit providing the input to the PLL supports either an external quartz oscillator connected to pins 
XTALN and XTALP or an external clock directly connected to the pin REF_CLK. The latter option could provide a better 
1/f noise perfor mance. The connected clock signal can be either a sine or square wave and can have amplitudes between 
1.2V -3.3V since the signal can be resistively divided. The input divider of the PLL itself can function with frequencies 
between 10 -40MHz. The default ext ernal clock signal should be at 20MHz.   
Important note -1: the user can only one of the two options! If the XTAL oscillator functionality is used, the REF_CLK 
input pins needs to be grounded, to avoid any disturbance. Vice versa if the REF_CLK input pin is used to supply an 
external reference clock of 20 MHz.  
Important note -2: the REF_CLK input is internally loaded with a 330 Ohm resistive divider.  
9.1.5 Optional connections  
The serializer chip also has 11 GPIO pins in the top row of the package and 4 pins for a slave SPI interface that was 
implemented for debugging purposes and normally do not need to be used by the customer.  
9.2 Operating the serializer  together with an image r 
The MAG -CXP00002 -NP serializer ASIC can in principle be used to interface with any imager which sends out parallel 
video data, as described in section 8.2. When it is GenICam  compliant, it really integrates seamlessly, because then the 
chip set can be considered as a single device from a user point of view. The latter is the case when the serializer is 
interfaced with the Magics Full HD imager product MAG -IMG001X1 -NC. As is already depicted in section 8.2, the 
functionality to reset and trigger the imager with hardware pins is already foreseen by the serializer. The register maps 
of both products are built in such a way that they interface perfectly by the use of the bootstrap FSM integrated in the 
serialize r. 
At the start -up of the devices, when connected to a CXP host/frame grabber, after a proper  connection  detection  and 
setup,  a connectionReset  is initiated  (see section 8.1). This means that the imager is reset by the serializer through the 
RESET_CAM pin. After that, the host will come and read the serializer to identify where all imager configuration registers

When a register in the address space 0x1000_0000 is addressed  by the user , the serializer knows which corresponding 
address to access in the imager to write or read the right data in the imager. So a ny access to this address range will 
initiate an SPI transaction to the imager through the dedicated imager master SPI. When the synchronization block 
notices that a setting is changed that also the serializer needs to be aware of, the setting is also copi ed to the internal 
registers of MAG -CXP00002 -NP. This results in the case that the user only needs to know of one register map, i. e. the 
one from the serializer.   
When a COTS frame gabber (CXP host) is used which implements the GenICam  standard, all of this is actually invisible 
to the customer. The frame grabber will come and read the address for each configuration setting  
(‘IIDC2EntryImager_*’ ) and will from then on use these configuration setting addresses. In the case of a custom  
implementation, it is however important to understand the mechanism. An example of an address access to the register 
map is given below.  
Assume the user want to modify the width of the imager (cfr. ‘ImageSize_Width’ ). This configuration register is present 
at address 0x7398 in the  MAG -IMG001X1 -NC imager. After start -up, this register is however copied to the serializer  at 
address  0x1000_1398 , since the address 0x6000 in the imager corresponds to the address 0x1000_0000 in the serializer:  
 0x7398 – 0x6000 + 0x1000_0000 = 0x1000_1398  
So if the user wants to access this register in the imager, he needs to access register 0x1000_1398 in the serializer, and 
the serializer will take care of the rest.

11. Register bank  
The register bank is built in such a way that it supports the GenICam  and IIDC2 standard when assembling the serializer 
in a camera. When used as a chip set with the imager product MAG -IMG001X1 -NC, a full GenICam  and IIDC2 compliant 
product can be built. As a result of this is the register map quite elaborate. However, quite some entries are constants 
and read only.  
There are different types of registers defined in the map, and they are listed below:  
- Constant : registers which hold a constant value and are used as information for the user or for the CXP host to 
understand the configuration of the device.  
- Configuration : configuration registers which are accessible to the customer.  
- Configuration_otp : configuration registers which are accessible to the customer, but which are programmed 
to a default state by Magics for optimal operation of the device.  
- Input : read -only values which get their input from somewhere else , such as analog status signals.  
- Trigger : these register values are used to trigger an action (e.g. connectionReset ) 
- State : type of configuration register which can be controlled by the user, but which can also be overwritten by 
the digital core .

refer to section 11.2  for the register settings which are not of the type constant  and to section 11.3  for the register  bits

exposure time s, etc.,  section 11.2  describes how these settings can be configured. However, typically these settings 
have a default value, but also a  corresponding  minimum and maximum value. These corresponding minimum and
